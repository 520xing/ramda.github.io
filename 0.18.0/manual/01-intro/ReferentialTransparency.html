Referential Transparency

Referential Transparency is a simple notion: we should be free to substitute an expresion's value for that expression without changing the behavior of our program.

There are several advantages to referetially transparent code. It simplifies analysis, and lets us -- or perhaps a compiler -- optimize parts that might otherwise be too difficult to do. But perhaps the greatest benefit is that it entirely removes the notion of sequencing, of time, from the analysis of our program. It doesn't matter which expressions are evaluated first; the answer will always be the same.

This sample is not referentially transparent:

var counter = 0;
var incrBy = function(n) {
  counter = counter + n;
  return counter;
}

This function will return different values when called again with the same parameters. It will also updates a global variable that might be used by other parts of the program. Both of these facts make it non-referentially transparent.6

There are two basic characteristics of referentially transparent functions: their output must be based only on the parameters supplied, and they must have no side effects. All of Ramda's functions are designed this way.

Output Based Only on Input

The function above was not pure because it depended not only upon the input parameter, n, but also upon the global variable, counter. That variable could change because of something else in the program unrelated to our own code. No Ramda functions do this. If we pass the same values to a Ramda function a second (or third, or seventy-fourth) time, we get the same result we did the first time.

There's a subtle point that could come across as confusing: functions that have been partially applied through curry, partial, or something similar do not lose this property.

So in this sample,

const add = R.curry((a, b) => a + b);
const add42 = add(42);

add42 is a referentially transparent function. Although thought of as a specialization of add, it involves a value stored in a local closure, and not a parameter, which makes it seem as though this might not be so transparent. But when viewed as a function in its own right, the value is not dependent upon any external state at all. The 42 is simply part of its logical definition.

No Side-effects

The other important feature of referentially transparent functions is that they perform no observable side-effects. Such functions cannot updates global state, cannot read or write from streams (including developer consoles), cannot update external systems such as a visible DOM.
